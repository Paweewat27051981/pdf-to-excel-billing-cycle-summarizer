# สรุปสถาปัตยกรรมระบบ (Architecture Overview) — สำหรับผู้รีวิวโค้ด

> ระบบคำนวณค่าเที่ยวรถร่วม + สรุปยอดตามรอบบิล
> นำเข้าใบกระจายสินค้า (PDF/Excel) → ตรวจทาน → คำนวณค่าเที่ยวรายรอบ (1–15 / 16–สิ้นเดือน) → สรุปรับสุทธิต่อทะเบียนรถ → ส่งออก Excel

---

## 1. เทคโนโลยี (Tech Stack)

| ชั้น | เทคโนโลยี |
|---|---|
| Frontend | React 19 + TypeScript, Vite, Tailwind CSS, lucide-react, SweetAlert2 |
| Backend | Node.js + Express (TypeScript รันด้วย tsx / build ด้วย esbuild) |
| ฐานข้อมูล | Firebase Realtime Database (+ cache ในหน่วยความจำ; fallback เป็น db.json) |
| AI อ่าน PDF | Google Gemini (`@google/genai`) |
| Excel | ExcelJS (ส่งออก), xlsx (นำเข้า) |

รันแอป: `npm install` → ใส่ค่าใน `.env.local` (ดู `.env.example`) → `npm run dev` (เปิด http://localhost:3000)

---

## 2. แผนผังไฟล์สำคัญ (อ่านตามลำดับนี้)

```
src/types.ts          ← (1) โครงสร้างข้อมูลทั้งระบบ — เริ่มอ่านจากที่นี่
src/calc.ts           ← (2) หัวใจการคำนวณ (pure functions ใช้ทั้ง client+server)
server.ts             ← (3) REST API ทั้งหมด
server-db.ts          ← (4) ชั้นจัดเก็บข้อมูล (Firebase + cache + write-through)
excel-import.ts       ←     อ่านใบกระจาย/ราคา/น้ำมัน จาก Excel (ไม่ใช้ AI)
src/excel-export.ts   ←     ส่งออกรายงาน Excel หลายชีต
src/App.tsx           ← (5) หน้าจอ UI ทั้งหมด (แท็บการทำงาน)
```

## 3. กระบวนการหลัก (Data Flow)

```
ผู้ใช้อัปโหลด PDF ──► /api/extract-pdf (Gemini) ──┐
ผู้ใช้อัปโหลด Excel ─► /api/import-excel ─────────┤
ผู้ใช้กรอกเอง ───────────────────────────────────┘
        │
        ▼
   หน้าตรวจทาน (Review)  ◄── /api/trips/preview (คำนวณโชว์ ไม่บันทึก)
        │  แก้ไข + เห็น warnings
        ▼
   /api/trips (POST) ──► เซิร์ฟเวอร์คำนวณใหม่จากข้อมูลดิบ + ข้อมูลหลักล่าสุด
        │                 + เปิดรอบอัตโนมัติตามวันที่ในใบ
        │                 + กฎเหล็ก: เลขใบกระจายห้ามซ้ำในสาขา (409)
        ▼
   บันทึกลง Firebase (เขียนราย record)
        │
        ▼
   Dashboard / รายงานต่อทะเบียน ──► Export Excel / พิมพ์ PDF
```

**หลักการสำคัญ:** ยอดเงินคำนวณ**ฝั่งเซิร์ฟเวอร์เสมอ** จากข้อมูลดิบ (extracted) ไม่เชื่อยอดที่ส่งมาจากเบราว์เซอร์

## 4. แนวคิดธุรกิจที่ต้องเข้าใจก่อนรีวิว (อยู่ใน `calc.ts`)

- **รอบบิล:** วันที่ 1–15 = รอบแรก, 16–สิ้นเดือน = รอบสอง (จัดเข้ารอบจากวันที่ในใบอัตโนมัติ)
- **ราคาเหมา vs ราคาชิ้น:** ปลายทางหนึ่งมีได้ทั้งคู่ ระบบเลือกอัตโนมัติ (จุดตัดจำนวน หรือเลือกยอดสูงกว่า)
- **กฎตัวหาร:** สินค้าชิ้นเล็กหลายชิ้นแปลงเป็นหน่วยคิดเงิน คำนวณแยกตามเลขใบรับสินค้า
- **งานพิเศษ:** เก็บสินค้าคืน (ครึ่งราคาชิ้น), Peat mass (ชิ้นละ 20/ราคาอำเภอ), ค่าเหมาบวกเพิ่มตายตัว
- **ราคาขั้นบันได / ราคาชุดหลายอำเภอ / กลุ่มราคาตามรถ / ราคาเฉพาะรอบ (override)**
- **สูตรรับสุทธิ:** `รวมค่าเที่ยว − หัก1% − ค่าน้ำมัน + รายได้เพิ่ม − รายการหัก`
- **หลายสาขา:** ทุก record มี `branchId`; แต่ละสาขาแยกข้อมูล/ราคา/รหัสผ่าน; HQ เห็นรวมทุกสาขา

## 5. การจัดเก็บข้อมูล (`server-db.ts`) — จุดที่ควรรู้

- อ่าน Firebase **ครั้งเดียวตอนบูต** → เก็บ cache ในหน่วยความจำ; ทุก `getDb()` ตอบจาก cache (ลดการดาวน์โหลด)
- คอลเลกชันใหญ่ (`tripDocuments`, `fuelEntries`, `deductions`) เก็บแบบ **id-keyed** เขียนราย record (`saveRecord`) → บันทึกเร็วคงที่
- คอลเลกชัน master อื่น ๆ ใช้ `saveDb()` = **เขียนทับทั้ง tree** (ระวังเรื่อง instance ซ้อนกันเขียนทับ)
- Firebase ตัด array ว่างทิ้ง → ต้อง normalize รูปร่างข้อมูลใน `ensureShape()` ทุกครั้งที่อ่าน

## 6. ไฟล์ที่ "ไม่ได้แนบมา" (เป็นความลับ — ต้องตั้งค่าเอง)

| ไฟล์ | ทำหน้าที่ |
|---|---|
| `serviceAccountKey.json` | กุญแจ Firebase Admin (ดู `.env.example` / docs ของ Firebase) |
| `.env.local` | `GEMINI_API_KEY`, `PORT`, ค่าอื่น ๆ |
| `db.json` | sandbox local (สร้างเองอัตโนมัติเมื่อไม่มี Firebase) |

> ถ้าไม่ใส่ Firebase key ระบบจะ fallback ใช้ `db.json` ในเครื่องได้ทันที (เหมาะกับการอ่าน/ทดสอบโค้ด)
